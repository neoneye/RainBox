## TL;DR

The assistant grew from a read-only ReAct skeleton into a **full personal-assistant
write surface**, each write **risk-tiered, traced, and reversible**:

- **Memory** — remember / forget (log-and-undo) / activate (confirm)
- **Skills** — propose candidate (log-and-undo) / activate (confirm)
- **Kanban** — move / complete / comment / create (log-and-undo)
- **Reminders** — schedule a one-shot message (confirm + dry-run preview)
- **Files** — propose an edit shown as a diff (confirm + dry-run, workspace-confined)

Plus: the **Phase 3 memory story is complete** (hybrid retrieval + an operator
profile block, in the assistant *and* the chat agents); a **`memory_sync` cron
trigger** keeps embeddings fresh; **`rainbox doctor`** is an operator health
check; and a uniform **version-guard pattern** means no reversible write can
clobber a state that changed underneath it. A real **production retrieval bug**
(unrelated queries retrieving the entire memory set) was found and fixed.

Plus, since: an **`/assistant` run inspector** (the inspection + approval slice of
the S7 dashboard) — a run-centric timeline with each write-intent inline and the
confirm/reject/undo + stop/redirect buttons, so confirm-tier writes finally have
a browser approval surface; an **`assistant_run_summarizer` agent** that digests
each completed run (trigger + obstacles + outcome) off the critical path; and
**`assistant_run` is now keyed by its `uuid`** (no integer id) — children
reference it via `run_uuid`, and the uuid is the one identifier shown, addressed
(`/assistant?id=<uuid>`), and logged.

**Not built (need your direction):** the *rest* of the runtime dashboard (S7 —
live streaming, kill/retry, in-flight visibility; the inspection + approval slice
shipped), the external-system / MCP adapters (S9/S10), the optional async profile
deriver
(S11), and a few small follow-ups (S12).

## How to verify (general)

All from the `source/` directory, using the project venv (the system `pytest`
lacks deps and the embedder env):

```bash
cd source
# 1. The whole story is tested, model-free, against the rainbox_claude sandbox DB:
./venv/bin/python -m pytest agents/ memory/ db/ skills/ tools/ user_profile/ webapp/ -q

# 2. Operator health check (capabilities, model groups, embedder, skills, MCP):
./venv/bin/python -m tools.doctor

# 3. Run the app (Flask webserver + idle supervisor) to use it live:
python3 main.py        # then open the chat UI it serves; see README for setup
```

**No shell needed for the health check:** once the app is running, the same
checks are at **`/doctor`** in the browser — a color-coded table plus a
copy-paste plain-text report (nav link after "Voice ▾"). The CLI (#2) is for
when the server isn't up.

Tests are pinned to `rainbox_claude` by `conftest.py` — they can never touch
production. A green suite is the primary evidence; the per-feature tables below
name the specific files so you can verify one area at a time.

## What's implemented — feature by feature

Each entry: **what it is · where · quality · verify (tests + manual) · use.**

### 1. Assistant ReAct loop + capability registry  *(skeleton predates this session; v2 PRs 1–10)*
- **What:** a bounded plan→act→observe loop with a durable per-step trace
  (`assistant_run`/`assistant_step`), and a code-owned capability registry
  (`CAPABILITIES`) that generates both the prompt catalog and dispatch.
- **Trace shape:** `assistant_run` is keyed by its **`uuid`** (no integer id); the
  child tables reference it via **`run_uuid`** (FK → `assistant_run.uuid`,
  CASCADE). Each logical step is **one mutable `assistant_step` row** — opened at
  `running` (durable before the action) and settled in place to `observed`/
  `failed`; terminal-only steps (`failed`-at-plan, `final`, `control`) are a single
  insert. Because the step is one addressable row, a write a step produces links
  back to it by `assistant_write_intent.step_uuid` (FK → `assistant_step.uuid`).
- **Where:** `agents/assistant.py`, `db/assistant.py`, `db/models.py`.
- **Quality:** solid, the foundation everything else plugs into. The model can only
  request enabled, prompt-exposed capabilities (enforced in `_validate_decision`).
- **Verify:** `pytest agents/test_assistant.py agents/test_assistant_fakes.py
  agents/test_capability_registry.py`. The fake-model seam (`scripted_decisions`)
  drives the loop deterministically. `tools.doctor` prints the live capability count.
- **Use:** add the `assistant` role to a `/chat` room and message it.

### 2. Phase 3 — user profile block + hybrid memory  *(this session + earlier)*
- **What:** `retrieve_memories_hybrid` (vector + Postgres full-text + entity boost,
  hard-filtered) and a query-independent **profile block** (a compact digest of
  active preferences/decisions/operator-facts), injected before the skills block.
- **Where:** `memory/retrieval.py`, `user_profile/`, `agents/chat_context.py`.
- **Quality:** good; the "filter before rank" contract (secret/expired/out-of-scope
  removed first) is shared and tested. **Now used by the chat agents too**, not just
  the assistant.
- **Verify:** `pytest memory/test_hybrid_retrieval.py user_profile/
  agents/test_chat_context.py agents/test_chat_memory.py`. Key tests assert: a
  stemmed full-text match recalls where token-overlap misses; secret/expired
  claims never surface; an unrelated query retrieves **nothing** (the bug fix);
  the profile appears before the memory block in the prompt.
- **Use:** seed memory (tell the assistant "remember that …", or `remember that …`
  in chat); ask a related question; the relevant facts + profile are injected.
- **Spec:** [`2026-06-20-phase3-user-profile.md`](2026-06-20-phase3-user-profile.md).

### 3. Embedding freshness + the `memory_sync` cron trigger  *(this session, S1)*
- **What:** embeddings are refreshed on the memory write path and pruned when
  stale; a seeded **enabled daily** in-process cron action (`memory_sync`)
  reconciles them so retrieval doesn't silently degrade to lexical-only.
- **Where:** `memory/embeddings.py`, `db/cron.py` (`cron_create_one_shot_message`
  pattern, the `memory_sync` action, the seeded `MEMORY_SYNC_CRON_JOB_UUID`).
- **Quality:** good; degrades gracefully when the embedder is down (warns, prunes).
- **Verify:** `pytest memory/test_embeddings.py db/test_cron_firing.py -k
  "memory_sync or refresh or prune or sync or seed"`. `rainbox doctor` shows the
  embedder status; the cron admin page (`/cron`) shows the seeded job.
- **Use:** automatic. Manually: `python -m tools.doctor` to confirm the embedder is
  reachable; the job runs daily at 03:15 local.

### 4. Write families + the log-and-undo ledger + confirm/dry-run  *(this session, S2–S5, S3)*
- **What:** five write families, two tiers:
  - **log-and-undo** (execute now, reversible): `remember`, `forget_memory`,
    `propose_skill`, `kanban_move/complete/comment/create`. Each lands a
    `completed` `assistant_write_intent` carrying an **inverse op**;
    `undo_write_intent` (and `POST …/undo`) replays it. An explicit `remember`
    creates an **active** memory (embedded immediately; no candidate step — that
    review tier is for future *inferred* memories). `forget_memory` is its mirror
    image: it rejects a memory by `memory_uuid` (from `query_memory`, which shows
    uuids) or by text (matching active **and** candidate memories) and executes
    immediately; undo (`reactivate_memory`) restores it.
  - **confirm** (propose → operator approves → execute): `activate_memory`,
    `activate_skill`, `set_reminder`, `edit_file`. `set_reminder`/`edit_file` use
    a **dry-run preview** (the resolved schedule / a unified diff).
  - **Internal inverses** (not model-invocable): `skill_delete`,
    `kanban_delete_task`, `reject_memory_candidate`, `reactivate_memory`.
- **Where:** `agents/assistant.py`, `agents/assistant_writes.py`, `db/assistant.py`,
  `db/cron.py` (reminders), `tools/workspace_policy.py` (file confinement).
- **Quality:** strong and consistent. **Version guards** mean no reversible write
  clobbers changed state: `edit_file` refuses if the file changed since the preview
  (base SHA); kanban undo refuses if the task moved since (`expect_column`); skill
  and memory undos refuse if the item was since activated (status guard).
  Confirm-tier writes can never execute inline (code-enforced).
- **Verify:** `pytest agents/test_assistant_writes.py agents/test_kanban_*.py
  agents/test_reminders.py agents/test_edit_file.py agents/test_skill_candidates.py`.
  These cover: undo restores prior state; confirm doesn't write inline; bad input
  fails at propose time; the version guards refuse stale writes; the model cannot
  invoke the internal delete/reject inverses.
- **Use (via `/chat`):** "mark the auth task done", "remind me Friday 09:00 to ship",
  "edit notes.md to add a TODO". The assistant **proposes** confirm-tier writes;
  approve via the write-intent endpoints (`/chat/api/assistant/write-intents/<uuid>/
  {confirm,reject,undo}`). `edit_file` only touches files under the workspace root
  (`/tmp/pp_workspace_shell`) and rejects traversal/`.env`/`.ssh`/symlink-escape.
- **Specs:** the `2026-06-20-s{2,3,4,5}-*.md` files under `docs/superpowers/specs/`.

### 5. `rainbox doctor` (CLI + browser page)  *(this session, S6)*
- **What:** a per-check health report (capabilities, model groups, embedder
  reachability, skills + unparseable-file lint, MCP servers), exits non-zero on
  failure. Available **two ways**: the CLI `python -m tools.doctor`, and the
  **browser page `/doctor`** (nav link after "Voice ▾") — a color-coded table plus
  a copy-paste plain-text report with a Copy button, re-runs on load.
- **Where:** `tools/doctor.py`, `skills.lint_skills`; the web view is
  `webapp/doctor_views.py` (route `/doctor`, nav in `webapp/core.py`).
- **Quality:** good, immediately useful (the embedder check is the recurring gap);
  the page means you never need a shell/SSH to see health.
- **Verify:** `pytest tools/test_doctor.py webapp/test_doctor_views.py`; then run
  the CLI (`python -m tools.doctor`) and/or open `/doctor` in the browser.
- **Use:** run it after setup or when something feels off — CLI when the server is
  down, the page once it's up.

### 6. `/assistant` run inspector  *(later; S7 inspection + approval slice)*
- **What:** a run-centric page — left pane lists recent runs (newest first, by
  start date, each with its summary + obstacle badge); right pane shows the
  selected run: a **Summary** block, then the run details (uuid + Copy, trigger
  message with an "open in chat" deep-link, the step timeline with each
  `assistant_write_intent` rendered **inline** under its step). The lifecycle
  actions are wired to the existing endpoints — **confirm / reject** a proposed
  write, **undo** a completed one, **stop / redirect** a live run — so confirm-tier
  writes finally have a browser approval surface (no more raw `curl`).
- **Where:** `webapp/assistant_views.py` (route `/assistant?id=<run-uuid>`, nav
  "Assistant"); reuses the write-intent + run endpoints in `webapp/chat_api.py`.
- **Quality:** good for inspection + approval; **read-only data** (raw edits stay in
  Flask-Admin). Live auto-refresh, kill/retry, and in-flight visibility are the
  remaining S7 work.
- **Verify:** `pytest webapp/test_assistant_views.py`; open `/assistant`.
- **Spec:** [`../superpowers/specs/2026-06-23-assistant-run-inspector-design.md`](../superpowers/specs/2026-06-23-assistant-run-inspector-design.md).

### 7. `assistant_run_summarizer` agent  *(later)*
- **What:** a dedicated agent that, **after** a run reaches any terminal state,
  digests it into `assistant_run.summary` `{trigger, obstacles[], outcome}` via one
  structured call — off the critical path (the assistant enqueues it and returns
  immediately; the supervisor runs it in its own process). `requires_structured_output`
  means `/agent_models` only offers it structured-output groups (the "structured
  override only" constraint, enforced by the existing page). Shown in the
  `/assistant` inspector.
- **Where:** `agents/assistant_run_summarizer.py`, registry in `agents/config.py`,
  `db.set_run_summary`, the enqueue in `agents/assistant.py`.
- **Quality:** good; graceful — until a model group is bound on `/agent_models` its
  runs fail and the run simply stays unsummarized; the assistant is never blocked.
- **Verify:** `pytest agents/test_assistant_run_summarizer.py`.
- **Spec:** [`../superpowers/specs/2026-06-23-run-summarizer-agent-design.md`](../superpowers/specs/2026-06-23-run-summarizer-agent-design.md).

### Steps

1. **Setup** (per README): create `rainbox_production` + `rainbox_claude`, install
   the venv, run Ollama with the embedder, configure at least one model group.
2. ✅ **Health:** `cd source && python -m tools.doctor` → expect all `✓` (or a clear
   `!`/`✗` telling you what's missing). This alone verifies capabilities, model
   groups, embedder, skills, MCP. (Or skip to step 3 and open **`/doctor`** in the
   browser — same checks, copy-paste-able, no shell.)
3. ✅ **Run:** `python3 main.py` (webserver on `http://127.0.0.1:5000`), open the
   chat UI, create a room, add the `assistant`. Visit **`/doctor`** (nav after
   "Voice ▾") to confirm health from the browser.
4. **Read path:** ask it something; confirm it plans→acts→replies and that the
   step trace renders in chat (expand the `debug-assistant` row — it shows each
   step's action, args, and observation).
5. **Memory + profile** *(execute side ✅; retrieval still to confirm):*
   - `remember that I prefer concise replies` → ✅ creates an active memory (the
     reply links to it on `/memory`).
   - **Now confirm it's actually retrieved:** in a *new* turn ask a related
     question (e.g. "how should you answer me?") and confirm the fact steers the
     reply **and** appears in the `debug-memory` trace row for that turn. This is
     the hybrid-retrieval + profile-block path, not just the write.
   - `forget that I prefer concise replies` → ✅ confirm it's gone (rejected on
     `/memory`, and no longer retrieved).
6. **Log-and-undo — verify the *undo*** (the apply side is ✅):
   - Ask it to move or complete a kanban task; confirm the board changed.
   - Find the write-intent uuid: it's in that turn's `debug-assistant` trace
     (the observation carries `write_intent_uuid`), or query it directly:
     ```bash
     # the live app runs on rainbox_production by default
     psql -d rainbox_production -c "select uuid, capability_name, state, created_at \
       from assistant_write_intent where state='completed' order by created_at desc limit 5;"
     ```
   - Revert it and confirm the board moves back:
     ```bash
     curl -X POST http://127.0.0.1:5000/chat/api/assistant/write-intents/<uuid>/undo
     ```
     Expect `{"ok": true, ...}`; the intent flips `completed → undone` and the
     task returns to its prior column.
7. **Confirm + dry-run write** *(the unexercised tier — no UI yet, approve via
   endpoint):*
   - Ask for a reminder (`remind me at 2026-06-24T09:00 to ship the release` —
     **absolute ISO-8601 only** today) or a file edit (`edit notes.md to add a
     TODO line`).
   - Confirm the assistant **proposes** rather than applies: its reply says the
     write *awaits your confirmation*, and the `debug-assistant` trace shows the
     **dry-run preview** (the resolved fire instant for a reminder; a unified
     diff for a file edit). Nothing has changed yet.
   - Grab the proposed uuid (trace observation `write_intent_uuid`, or the DB):
     ```bash
     psql -d rainbox_production -c "select uuid, capability_name, preview_text \
       from assistant_write_intent where state='proposed' order by created_at desc limit 5;"
     ```
   - **Approve and confirm it now applies:**
     ```bash
     curl -X POST http://127.0.0.1:5000/chat/api/assistant/write-intents/<uuid>/confirm
     # to decline instead:
     curl -X POST http://127.0.0.1:5000/chat/api/assistant/write-intents/<uuid>/reject
     ```
     After `…/confirm`: the reminder appears as a one-shot job on `/cron`; the
     file edit is written under the workspace root (`/tmp/pp_workspace_shell`).
     The same `confirm`/`reject` flow approves `activate_memory` and
     `activate_skill`.
8. **Version guard** (do this with `edit_file`):
   - Ask for a file edit so a proposal is recorded (don't confirm yet).
   - Change that file underneath — e.g. `echo "drift" >> /tmp/pp_workspace_shell/notes.md`.
   - POST `…/confirm`. It must **refuse** (the response text says the file
     *changed since the preview; not applying*) rather than clobber — because the
     proposal pinned the file's `base_sha`. Re-propose to pick up the new content.
9. **Cron:** check `/cron` shows the seeded **`Memory embedding sync`** job
   (enabled, daily 03:15 local).

If steps 4–9 behave as described, the full implemented surface — read, both
write tiers, undo, the version guard, and the maintenance cron — is working
live. Any divergence is most likely model-prompting (the local model choosing
odd actions) rather than the mechanics, which the tests pin.

## Rough edges (known, acceptable, documented)

- **Char-based prompt budgets**, not tokenizer-aware. Fine for now; crude for very
  small local context windows (S12).
- **`edit_file` uses full-content replacement** (the model sends the whole new file)
  — robust but token-heavy for large files; line-range patches are a follow-up.
  Confined to `/tmp/pp_workspace_shell`; the sensitive-name denylist was sized for
  reads and is exact/case-sensitive (e.g. `.git/config`, `.ENV` aren't covered).
- **Reminders take an absolute ISO-8601 time**; "Friday 9am" needs current-time
  injection + a parser (follow-up). UTC-default for naive times (the dry-run
  preview shows the resolved instant, so a wrong tz is visible before confirm).
- **`retrieve_memories` (token-overlap)** is now production-unused but kept as the
  lexical baseline/contrast in tests.
- **No automated revert of an applied confirm-tier write** (file edit, activation);
  the operator approved the exact change, and undo exists only for log-and-undo.
- **`rainbox doctor`** has both a CLI (`python -m tools.doctor`) and a browser
  page (`/doctor`, nav after "Voice ▾"), but probes shallowly (no migration-drift /
  pgvector-presence / per-agent model-group validity checks).
- **Profile facts** = active claims with a non-null subject; "subject referring to
  the operator" is approximated, so an arbitrary subject'd fact can surface.

## Not implemented — and why

- **S7 — runtime dashboard UI (partial).** The **inspection + approval slice
  shipped** as the **`/assistant`** page (timeline + confirm/reject/undo +
  stop/redirect). *Still not done:* **live** in-flight visibility (PID / current
  step / heartbeat age), live streaming (SSE / LISTEN-NOTIFY) instead of a manual
  reload, and the **kill** (watchdog) / **retry** (re-enqueue) controls. The
  `/assistant` page is the pattern to grow these from.
- **S9 → S10 — external-system / MCP adapters.** Activate `Capability.adapter` to
  route the assistant through narrow read-only (then, later, write-capable) MCP
  tools. *Not done:* routing the assistant to external tools is a real capability
  and security expansion — *which* servers/tools and the read/write boundary are
  operator decisions, not routine forks. (MCP infra exists; the `adapter` field is
  the dormant seam.)
- **S11 — async profile deriver.** A background agent proposing inferred profile
  facts. *Not done:* explicitly optional in v2 — build only if the one-shot profile
  proves stale in practice.
- **S12 leftovers.** Read-only contradiction surfacing (low payoff today — most
  claims lack structured subject/predicate/object); tokenizer-aware budgeter;
  promoting the `eval_case` regression layer; project-scoped profile facts (needs a
  project key on the turn). Small, opportunistic.
- **Admin page for the write-intent queue:** the `confirm/reject/undo` endpoints
  exist but there's no web surface to review/approve pending writes yet. (The
  doctor admin page **is** now built — see `/doctor`.)
